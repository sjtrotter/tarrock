# _Project — Tarrock game code root

All first-party Tarrock content lives under `Assets/_Project/`, keeping it isolated from
Unity template assets and imported packages. This is milestone **M1 phase 1**: services,
data model, tests, and editor tooling only — no gameplay yet.

Layout and conventions are owned by
[`docs/design/technical.md`](../../../docs/design/technical.md). This README maps the
on-disk tree to that spec and records which folders exist now versus which arrive with
their first content. (Empty folders can't be committed through Unity — a `.gitkeep` would
import as an asset — so folders are created only when the first file lands in them.)

## Assemblies present this phase

- `Tarrock.Core` — no project references.
- `Tarrock.WorldState` — references `Tarrock.Core`.
- `Tarrock.Regions` — references Core + WorldState. First content: `PlayerSpawnPoint`,
  `LeapOfFaithTrigger`, `InteractionMarker`, and the `CliffMarkerIds` constants surface,
  used by the Cliff greybox.
- `Tarrock.Editor` — Editor-only; references Core + WorldState + Regions.
- `Tarrock.Tests.EditMode` — Editor-only; references Core + WorldState + TestRunner.

The per-feature assemblies for Quests, Combat, Trumps, Dialogue, and UI (`Tarrock.Quests`,
etc. — see technical.md §Assembly definitions) are intentionally **not** created yet: an
asmdef with no scripts produces an empty-assembly warning. Each is added in the same
change as its first script.

## Editor tools (run from the Unity menu bar)

- **Tarrock/Setup/Create Core Scenes** — creates `Bootstrap.unity` (a `CompositionRoot`
  object) and `Core.unity` (`Services`, `UI Root` roots). Idempotent.
- **Tarrock/Setup/Generate WorldState Definitions** — creates one `WorldStateDefinition`
  asset per world-state id under `Data/WorldStates/`. Idempotent.
- **Tarrock/Validate/World State Ids** — errors on any drift between the definition assets
  and the `WorldStateIds` constants (the docs↔data check technical.md promises).
- **Tarrock/Setup/Generate Cliff Greybox** — builds `Scenes/Regions/Cliff.unity`, the MQ00
  greybox (campfire → old campsites → dead tree knoll → standing stones → Blank ambush →
  Waystation → cliff edge), and the shared `Art/Greybox/` materials. Idempotent — aborts if
  the scene already exists.
- **Tarrock/Setup/Regenerate Cliff Greybox (Overwrite)** — deletes the existing Cliff scene
  (if any) and rebuilds it from scratch.

## Note on `CompositionRoot` placement

`CompositionRoot` lives in the `Tarrock.WorldState` assembly rather than `Tarrock.Core`,
even though it is the bootstrap type. It must construct `WorldStateService` / `SaveService`
(which live in WorldState) while also referencing `StringEventChannel` (in Core). Housing it
in Core would force Core to reference WorldState, which already references Core — a cyclic
asmdef dependency. WorldState is the lowest assembly that can see both. See the type's own
doc comment for detail.
